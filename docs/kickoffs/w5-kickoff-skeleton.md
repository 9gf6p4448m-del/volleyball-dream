# W5 Kickoff 骨架 — 逐出機制

> 用途：交給 Claude Code，請其依此骨架＋W4 完整快照產出 `w5-kickoff.md` 決策底稿。
> 格式沿用 `phase3-kickoff.md`：每題附選項與 Fable 推薦，Sawmah 逐題拍板後回填、開工。
> **現況基準**：W4 完整快照（檔名待填）。
> **前置**：Sawmah 已實際試玩一輪招募，試玩觀察須寫入 kickoff 開頭。

## 產出前必答（寫進 kickoff 的「本輪前提」）

- 名冊上限 10 的壓力是否真的出現？（沒滿＝優先級下調、範圍縮小；第二賽季即卡滿＝剛需）
- W4 招募實際交付了什麼、留下哪些接口（依快照列表，勿依本骨架臆測）
- `updateTrust` 目前是否仍為零呼叫狀態

## 決策題（請 Claude Code 各附 2–3 選項與推薦）

1. **逐出成本強度**——trust 全體下修幅度？是否依 persona 分歧（好友掉更多）？是否有賽季內次數上限？
2. **不可逆的實作邊界**——逐出者永不回歸、招募條件同時失效，這條是否有例外（如條件已達成未領取）？
3. **確認流程形式**——二次確認的呈現方式（打字確認／長按／單純二次點擊），以及警告文案要揭露多少後果
4. **`career.scouting` 的 expelled 結構**——與現有宿敵記憶共用欄位還是另立？需存哪些欄位供 Phase 4 回歸對戰使用
5. **佔位台詞規格**——幾行、什麼語氣、如何在程式內標記待命名工程重寫（建議統一標記便於日後 grep）
6. **測試範圍**——名冊／trust 單元測試的最小集合

## 明確排除（不在 W5 範圍，kickoff 須複述以防範圍蔓延）

- 逐出當下的告別對話正式版 → **命名工程同批**（名字與 persona 未定寫不出好台詞）
- 個性化台詞差分 → 命名工程同批
- 轉學後回歸對戰＋賽前對話 → **Phase 4**
- 強化的敘事化（變強原因須為「當初逐出他的理由的反面」）→ Phase 4
- 自由球員市場 → Phase 4 業餘章節專門開一輪

## 硬性約束（沿用）

sim 純核心零 three.js/DOM、決定論、Intent 唯一輸入；手機 60 FPS；PWA 可包殼；
新系統補自動化測試；本輪**不動模擬核心**（若發現必須改 sim，先回報再議）。
