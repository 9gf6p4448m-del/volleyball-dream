## 決策題

### 1. 球員招募（Sawmah 已初步認可方向：條件制，非轉蛋）
- 擊敗對手後，達成**明示條件**＝該隊球員確定來投（越強條件越苛：勝場數/
  表現門檻/特定壯舉如「攔死他們王牌 3 次」）；來投演出做成開箱級儀式感
- 待定細節：a) 每隊開放 1 人還是多人？ b) 同場雙候選擇一（種子決定論）要不要？
  c) 招募失敗的條件可否跨賽季累積？
- 推薦：每隊 1 名招牌球員、無雙候選（首版單純）、條件跨賽季累積。

### 2. 名冊系統（招募的前提）
- 隊友具名＋個性化屬性＋各自成長曲線；招募球員帶原隊參數 DNA
- 待定：隊友成長是自動（表現驅動）還是玩家分配？板凳幾人？
- 推薦：隊友自動成長（玩家點數只管自己——保持主角感）、名冊上限 10。

### 3. 先發編排（啟用 7.7 驗證器）
- 賽前排 6+1 先發與輪轉起點；排錯陣＝真實輪轉犯規（驗證器已在等）
- 推薦：**做**。招募進來的人要有地方用；這是名冊的出口。

### 4. 賽季制／二周目
- **A. 線性多賽季**：奪冠/止步後進入下一屆（對手升級、宿敵記憶延續）
- **B. NG+**：帶著能力重開整條錦標賽、對手池隨機化（設計審查首推）
- **C. 兩者**：先 A 後 B
- 推薦：**A 起步**（招募與名冊需要的舞台）；對手池加 2-3 支新隊防重複感。

### 5. 決賽平衡終調
- 治具基準：決賽 23%（AI 代打）；等 Sawmah 真人一輪數據後定調
- 選項：天鷹 72→70／jump bias 微降／維持魔王（推薦先維持，數據說話）

### 6. 技術債第 0 項（開工前先還）
- main.js runMatch 拆分（660 行）＋geoCharacter draw calls（InstancedMesh）＋
  bundle 分割——Phase 3 要在上面疊名冊 UI，地基先穩
- 推薦：Phase 3 第一週專職還債，不與新功能並行。
