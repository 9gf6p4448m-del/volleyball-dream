## 1. Phase 1 範圍（垂直切片＝一場能玩完的球）

**要做**：
- 單局 **25 分 rally point**、完整 **6v6 規則與輪轉**（每球得分、觸球上限 3 次、後排攻擊限制、輪轉/換發）
- 玩家控主攻手一人，**五動作操作全套**（發/接/舉/扣/攔），觸控＋滑鼠統一向量邏輯
- **混合視角切換**（發球/扣球第一人稱，接發/防守/舉球肩後第三人稱）
- **11 人回合 AI 狀態機**（我方 5＋對方 6）——本階段地基級任務
- **資料層 v1** 定型（屬性＋技術＋身高時變＋信任參數預留）
- **視線欺敵原型**（第一人稱扣球視線方向餵攔網 AI，看左打右可行）
- **公平線索**：球的即時影子、鏡頭取景可讀性、擊球音效差異（全手動走位、無落點圈的前提）

**不做**（Phase 2 以後）：其他位置的操作體系、創角/劇情、訓練循環、聯賽賽程、多人連線。

## D 區 — 地基級決策（討論先鎖這區，返工代價最大）

### D1. 資料層 v1 結構（design-brief：「數據層第一天建對，後續章節只長內容不改結構」）

Phase 1 的球員只有一個玩家＋一堆 AI，但**資料結構現在就要定成能長到 Phase 3/4 的形狀**。以下是草案，請討論欄位取捨與命名：

```
Player = {
  id, name, teamId, positionRole,          // positionRole: 'outside'|'setter'|'libero'|'middle'|'opposite'
  height: { current, timeline },            // 時變屬性：current 當前身高(m)；timeline 預留成長曲線(Phase 3 用)
  attributes: {                             // 屬性層(0–100)，訓練提升
    jump, power, reaction, stamina, speed, control // 彈跳/力量/反應/體力/移速/控球
  },
  techniques: {                             // 技術層：解鎖與熟練度(Phase 2+ 長內容)
    // 例: jumpFloatServe:0, backRowAttack:0, tip:0 ... Phase 1 先放主攻手會用到的幾項
  },
  trust: { fromSetter: 50 },                // 信任參數預留(Phase 3 信任即戰術掛鉤)，Phase 1 給預設值不驅動邏輯
}
```
**待拍板**：
- (a) 屬性清單就這 6 項夠不夠？身高「影響攔網高度/扣球點/防守範圍」是**從 height.current 即時推導**（建議，避免資料重複），還是另存衍生欄位？
- (b) `trust` 這種 Phase 3 才啟用的欄位，Phase 1 要不要先進結構（我傾向要，避免之後改結構動到存檔相容性）？
- (c) 存檔格式：Phase 1 有沒有存檔需求？還是先純記憶體、Phase 3 創角時才定序列化？

### D2. 比賽狀態機 API（模擬核心的「規則流程」層，與 D3 的「球員行為」層分開）

一個 rally 的生命週期，寫成模擬核心裡的純狀態機（決定論、固定步長）：
```
SERVE → BALL_IN_FLIGHT → RECEIVE(一傳) → SET(二傳) → ATTACK(三擊) → OVER_NET
  → (對方) DIG/BLOCK → ... → DEAD_BALL(落地/出界/觸網/四擊/踩線)
  → SCORE_JUDGE(判給哪隊) → ROTATE_OR_HOLD(換發則輪轉) → 次一 rally
```
**待拍板**：
- (a) 這層 API 的邊界——它只吐「比賽事件」（誰得分、輪轉、死球原因），把「球員怎麼動」全交給 D3？（我強烈建議這樣切，這正是鐵律 1 的落地方式）
- (b) 觸球計數、後排攻擊違例、輪轉位置違例……Phase 1 要實作到多完整？（建議：得分/觸球上限/輪轉/後排攻擊限制這四條先做齊，「位置錯誤」等細則可延後並在文件標明）
- (c) 25 分制的收尾：先做「25 分且領先 2 分」的標準，還是 Phase 1 先簡化成「先到 25」？

### D3. 回合 AI 狀態機架構（design-brief 點名的「工程最大難點・地基級」）

11 個 AI 要打出像樣的排球：接發站位、判斷誰接、舉球分配、助跑起跳、攔防佈陣。**單機隊友／對手／未來多人補位共用同一套**。
**待拍板**：
- (a) 分層：**個體行為狀態機**（單一 AI：待命→判來球→移動到位→執行動作→回位）＋ **球隊協調層**（誰 call
  球、接發陣型、攔防佈陣、舉球分配決策）——這個雙層切法認不認可？
- (b) 「誰接球」的仲裁怎麼定（責任區 + 最近者 + 呼叫優先）——這是排球 AI 最容易打架/互讓的地方，要先定原則。
- (c) 難度：Phase 1 對手 AI 要多強？（建議先做「站得住、接得起、打得過來」的堪打等級，強度調參延後）
- (d) AI 決策也走固定步長、決定論嗎？（我強烈建議是——否則多人連線的同步前提破功）

### D4. 統一輸入抽象（承 vow3d 手勢系統；鐵律 1「未來連線餵網路指令」的關鍵）

> 「承 vow3d 手勢系統」指的是要繼承的**精神**（design-brief 原話）：**觸控與滑鼠共用同一套向量邏輯**
> ——同一個手勢（拖曳/點按）在手機與桌機產生等價的向量輸入，遊戲邏輯不分裝置。不需要 vow3d 的實作細節，
> 這裡要定的是排球版的中性意圖結構。

design-brief：五種動作 = **走位＋時機＋方向**三要素的不同配比。輸入層應把觸控/滑鼠都翻譯成一個**中性意圖結構**餵給模擬核心：
```
Intent = { move: vec2, action: 'serve'|'receive'|'set'|'spike'|'block'|null,
           aim: vec2, gaze: vec2, timing: 0..1 }   // gaze 供視線欺敵；timing 定擊球質量
```
**待拍板**：
- (a) 這個 Intent 形狀認不認可？未來網路封包、AI 決策、玩家輸入是否都產生同型 Intent（我建議統一，AI 和玩家走同一條指令管線）？
- (b) 觸控與滑鼠如何映射到同一套向量（拖曳＝aim＋timing？點按＝action？）——這題與 H1 連動，可一起談。

## H 區 — 玩法手感決策（品味／試玩題：給方向即可，靠試玩調，不在此拍死）

> 以下每題我**只列選項與取捨**，不替你選。手感是靠手指驗證的，弱模型的品味不可靠——這區的決定權在你和 Opus 討論後的判斷，
> 且 design-brief 已明說「具體按法在 Phase 1 kickoff 定案並以試玩調整」。

### H1. 五動作各自的操作（走位＋時機＋方向的配比）
| 動作 | design-brief 定的性質 | 選項/取捨 |
|------|----------------------|-----------|
| 發球 | 拖曳瞄準＋時機定質量 | 拖曳長度=力量 vs 拖曳方向=落點；跳發時機條 |
| 扣球 | 拖曳瞄準＋時機＋**視線欺敵** | 助跑自動 vs 手動；起跳時機窗；擊球點方向 |
| 接發/防守 | 吃預判與到位 | 走位到位即自動接 vs 需再一個確認鍵定方向 |
| 舉球 | 玩「分配給誰」 | 方向盤選攻擊手 vs 快捷鍵位；二傳質量條 |
| 攔網 | 預判與到位 | 起跳時機窗；手型/方向是否可控 |

**主攻手的「舉」怎麼碰到**：真實排球主攻手很少舉球（那是舉球員的事）。要讓玩家五動作全體驗，需設計觸發情境
——例如「舉球員接了一傳、或一傳直接過網需主攻手救球二傳」。**這題要定**，否則五動作缺一角。

### H2. 混合視角切換（發/扣第一人稱 ↔ 其餘第三人稱）
- 切換的**觸發**：由比賽狀態自動（輪到你發/你要扣就切一人稱）vs 玩家可手動覆寫？
- 切換**瞬間的方向感銜接**（design-brief 標為實作重點）：硬切 vs 短過場運鏡 vs 保持羅盤朝向。
- 這題最終「以試玩手感驗證，保留調整空間」（design-brief 原話）。

### H3. 視線欺敵原型（第一人稱扣球獨有・視角賣點轉核心玩法）
- 讀取：玩家一人稱「看向哪」餵給攔網 AI。**看哪打哪最穩；看左打右可騙開攔網，代價是難度＋失誤率上升**。
- **待定**：視線與實際擊球方向的夾角，如何換算成「騙過攔網的機率」與「自己失誤率上升」的曲線？（這是這機制成不成立的核心數值，建議 Phase 1 先做一條可調的簡單曲線，試玩再修）

### H4. 全手動走位的公平線索（無落點圈的前提，design-brief 最硬核刻度）
落點圈**不畫**。但現實存在的線索必須到位：
- **球的即時影子**（Phase 0 已有貼地圓影原型，可沿用強化）
- **鏡頭取景可讀性**（第三人稱的取景角度要能判斷來球深淺）
- **擊球音效差異**（大力扣 vs 輕吊 vs 一傳，聲音要能分辨）
- **待定**：這三個線索 Phase 1 各做到什麼精細度算「公平」？
