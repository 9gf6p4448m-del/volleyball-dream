## D 區 — 地基級決策【已定案】

### D1. 資料層 v1 結構 ✅

**屬性層擴為 8 項**（原 6 項不足以表達排球核心維度）：
`jump / power / reaction / stamina / speed / control / serve / block`
- 新增 `serve`：發球技術性強，與 power 不同源。
- 新增 `block`：攔網時機/判讀，與 reaction/jump 不同源。

**身高影響即時推導**：攔網高度、扣球點、防守範圍全部從 `height.current` 即時計算，**不另存衍生欄位**，避免資料重複與不同步。

**`trust` 與 `height.timeline` 現在就進結構**：空欄位成本近零，Phase 3 才加會動到存檔相容性。

**positionRole 拆成 natural / current**：`naturalRole`（依身高建議的位置）與 `currentRole`（玩家實選）分離，為 Phase 3 成長期身高的「建議 vs 選擇」敘事張力預留。

**存檔**：Phase 1 純記憶體、不做存檔 UI，**但序列化格式現在就定**——`Player` 的 JSON 序列化/反序列化寫成純函式並測試，Phase 3 接存檔時結構已驗證，不回頭改形狀。

定案結構：
```js
Player = {
  id, name, teamId,
  naturalRole,   // 依身高建議：'outside'|'setter'|'libero'|'middle'|'opposite'
  currentRole,   // 玩家實選（同上列舉）
  height: {
    current,     // 當前身高(m)，攔網高度/扣球點/防守範圍即時由此推導
    timeline,    // 成長曲線，Phase 3 用；Phase 1 留空
  },
  attributes: {  // 0–100，訓練提升
    jump, power, reaction, stamina, speed, control, serve, block,
  },
  techniques: {  // 解鎖與熟練度，Phase 2+ 長內容
    // Phase 1 先放主攻手會用到的：spike, jumpServe, block, receive, emergencySet
  },
  trust: { fromSetter: 50 }, // Phase 3 信任即戰術掛鉤；Phase 1 給預設值、不驅動邏輯
}
```
序列化：`serializePlayer(p) -> json` / `deserializePlayer(json) -> Player`，純函式，需單元測試往返一致（round-trip）。

### D2. 比賽狀態機 API ✅

**只吐比賽事件、不管球員怎麼動**（硬切）。狀態機是裁判不是教練，球員行為全歸 D3。這是鐵律 1 的落地。

事件流：
```
SERVE → BALL_IN_FLIGHT → RECEIVE(一傳) → SET(二傳) → ATTACK(三擊) → OVER_NET
  → (對方) DIG/BLOCK → ... → DEAD_BALL(落地/出界/觸網/四擊/踩線)
  → SCORE_JUDGE(判給哪隊) → ROTATE_OR_HOLD(換發則輪轉) → 次一 rally
```
狀態機對外只輸出事件：得分方、輪轉、死球原因、觸球計數。

**Phase 1 規則做齊四條**：得分 / 觸球上限(3) / 輪轉換發 / 後排攻擊限制。
**延後**：輪轉站位違例等細則 → 程式碼標 `// TODO Phase 2`（每球必觸發的是前四條、是骨架；站位違例是細則，缺了不影響打完一局）。

**收尾做滿 deuce**：「25 分且領先 2 分」完整實作，不簡化成「先到 25」。deuce 是 rally point 的靈魂，邏輯僅數行，一次做對、避免之後改測試。

### D3. 回合 AI 狀態機架構 ✅

**雙層架構**：
- **個體行為狀態機**（單一 AI）：待命 → 判來球 → 移動到位 → 執行動作 → 回位
- **球隊協調層**：誰 call 球、接發陣型、攔防佈陣、舉球分配決策

單機隊友／對手／未來多人補位**共用同一套**——這是唯一能讓三者共用成立的結構。

**「誰接球」仲裁優先序**（定死，避免打架）：
1. **責任區**：每個 AI 有主責區，來球落點在誰的區就是誰。
2. **呼叫鎖定**：某 AI 判定接球即發 "call" 鎖定該球、其他人退讓（模擬喊球）。call 一發出即進入**不可撤銷窗口**，杜絕兩人同撲、也杜絕互讓。
3. **最近者僅為 tiebreak**：落點在責任區交界時比距離；仍相等比固定 ID 序。

核心原則：**寧可有人搶錯，不可兩人互讓**——互讓（球落地沒人動）是排球 AI 最醜的破綻，比搶錯更傷沉浸感。

**決定論、固定步長，強制**：所有 AI 決策是 `(狀態, 固定步長 tick) → 決策` 的純函式。一旦吃浮點不確定或幀率依賴，多人同步前提破功、Phase 5 直接死，無妥協空間。

**難度＝堪打等級**：對手 AI 站得住、接得起、打得過來，強度調參延後。但**堪打≠弱智**——接發不能漏、舉球不能亂傳，否則垂直切片驗不出真實手感。堪打＝動作正確但不玩心理戰、不打刁鑽落點。

### D4. 統一輸入抽象 ✅

**Intent 統一**：玩家 / AI / 未來網路封包產生**同型 Intent**，模擬核心只認 Intent、不知來源（手指/AI/網路）。這條做對，Phase 5 多人是「換 Intent 來源」而非重寫。

定案結構（在草案上加 `playerId` 與 `tick`）：
```js
Intent = {
  playerId,   // 多人/多AI 時模擬核心需知是誰的意圖；須對得上 Player.id
  move: vec2, // 走位方向 -1..1
  action: 'serve'|'receive'|'set'|'spike'|'block'|null,
  aim: vec2,  // 瞄準/落點方向
  gaze: vec2, // 視線欺敵；僅一人稱扣球有效，其餘等同 aim
  timing: 0..1, // 擊球質量
  tick,       // 決定論：綁定此 Intent 生效的固定步長 tick
}
```

**觸控/滑鼠映射原則**（細節與 H1 綁定）：同一手勢在兩裝置產生等價向量——拖曳＝`aim`+`timing`、點按/放開＝`action` 觸發、長按＝`timing` 蓄力。

**跨題骨架提醒**：`Intent.playerId` 對應 `Player.id`（D1）、AI 與玩家共用同一條 Intent 管線（D3）、全走固定步長決定論——D1/D3/D4 是同一決定論骨架的三面，實作當一組做，勿分開。

## H 區 — 玩法手感決策【已定案・標示可調參】

### H1. 五動作操作 ✅

| 動作 | 定案操作 |
|------|----------|
| **發球** | 拖曳方向＝落點、拖曳長度＝力量、放開瞬間位置＝時機質量。跳發多一段起跳時機窗（早踩線/晚則觸球點低）。Phase 1 只給大力跳發一種，飄球為技術層解鎖（Phase 2+）。 |
| **扣球** | **助跑自動、起跳手動**。玩家專注：何時起跳（時機窗）＋往哪打（aim）＋看哪（gaze）。助跑手動會分掉視線欺敵的注意力預算，故自動。起跳時機窗決定擊球點高度。 |
| **接發/防守** | **走位到位即自動接，方向由到位時身體朝向決定**，不加確認鍵（一傳是反射不是瞄準）。到位品質影響球質：站定接 vs 撲救接 output 不同——走位吃預判的體現。 |
| **舉球** | **方向盤選攻擊手**（拖曳指向要傳的人），二傳質量由到位程度決定。方向盤為 Phase 3「信任影響分配」預留視覺語言（信任低時某方向盤選項「卡手」）。 |
| **攔網** | 起跳時機窗＋手型方向可控（aim 決定封哪條線）。與扣球視線欺敵對戲：封的方向＝讀的方向，被騙＝封錯邊。 |

**主攻手怎麼碰舉球**：用**「一傳過網救球」情境**——一傳品質差、球直接往對方場飛或高球需處理二傳時，最近的非舉球員（含主攻手）觸發救球式二傳。真實排球場面、天然低頻，符合「五動作全體驗但不喧賓奪主」。**不為湊數硬塞主攻手常態舉球。**

### H2. 混合視角切換 ✅

- **觸發：比賽狀態自動，不給手動覆寫**。輪到你發/進入扣球起跳即切一人稱，其餘三人稱。手動覆寫會破壞設計意圖並增輸入複雜度，Phase 1 先驗自動切。
- **方向感銜接：保持羅盤朝向＋極短過場運鏡（3–5 幀）**。切換瞬間鎖定「你面向球網」為共同參考，用 3–5 幀把鏡頭拉到新位置。關鍵：**切換前後「網在哪」不變**。
- 【試玩必調】銜接幀數與運鏡曲線為可調參，結構不變。

### H3. 視線欺敵數值曲線 ✅

Phase 1 做一條**可調的線性＋平方曲線**：
```
θ = 視線方向與實際擊球方向的角度差（0°＝看哪打哪）

騙過攔網機率   = min(θ / θ_max, 1) × deceive_gain     // 線性
自身失誤率增量 = (θ / θ_max)² × error_gain            // 平方

初值：θ_max = 45°, deceive_gain = 0.7, error_gain = 0.5
```
設計意圖藏在形狀差：**騙敵線性、失誤平方**。小角度偏移＝甜蜜區（騙敵收益高、失誤代價低）；大角度＝失誤平方級暴衝（高風險高回報）。
攔網 AI 讀 `gaze` 方向佈署封阻、實際球走 `aim`，兩者夾角即 θ，落地判定時結算。
【試玩必調】三參數全開成可調常數，調的是「甜蜜區多寬、極限多危險」。

### H4. 公平線索精細度 ✅

- **球影**：貼地圓影，**大小/透明度隨球高變化**（越高越大越淡、越低越小越實）。無落點圈下判斷落點的主力線索，影子收縮速度＝玩家讀「還有多久落地」的依據。Phase 0 原型加上「高度 → 尺寸/透明度」映射。
- **鏡頭可讀性**：Phase 1 門檻＝**第三人稱時鏡頭略升高俯角**，讓進攻方向縱深不至於看不出遠近。先固定一個「看得出深淺」的角度，不做動態運鏡。
- **擊球音效**：**三種音色可辨即達標**——大力扣（爆裂）/ 輕吊（悶短）/ 一傳墊球（脆彈）。差異大到閉眼能分。Phase 1 不做空間音效，先做音色可辨。

**H4 驗收主觀判準**：三線索合起來要讓「一個熟悉排球的人不看落點圈也能接到球」——試玩時找會打排球的人測。

## 3. 結論回填區【已填，即實作依據】

- **D1 資料層**：屬性 8 項（+serve/+block）；身高影響即時由 height.current 推導不另存；trust 與 height.timeline 現進結構；positionRole 拆 naturalRole/currentRole；Phase 1 純記憶體但序列化純函式現定並測。
- **D2 比賽狀態機**：只吐比賽事件、球員行為歸 D3；四條規則做齊（得分/觸球上限/輪轉/後排攻擊限制）、站位違例標 TODO Phase 2；deuce 完整做。
- **D3 回合 AI 架構**：雙層（個體行為＋球隊協調）；誰接球＝責任區→呼叫鎖定(不可撤銷窗口)→最近者 tiebreak→ID 序，原則「寧搶錯不互讓」；決定論固定步長純函式強制；難度堪打等級（正確但不玩心理戰）。
- **D4 統一輸入抽象**：Intent 玩家/AI/網路同型；加 playerId 與 tick；映射原則拖曳=aim+timing、點按=action、長按=蓄力。
- **H1 五動作操作**：發=拖曳落點+長度力量+放開時機；扣=助跑自動起跳手動+時機窗+aim+gaze；接=到位自動接、品質影響球質；舉=方向盤選人（為信任系統預留）；攔=時機窗+手型方向；主攻手舉球用「一傳過網救球」情境觸發。
- **H2 視角切換**：狀態自動切無手動覆寫；保持羅盤朝向+3–5 幀過場運鏡（幀數可調）。
- **H3 視線欺敵數值曲線**：騙敵線性、失誤平方；θ_max=45°, deceive_gain=0.7, error_gain=0.5，三參數可調。
- **H4 公平線索精細度**：球影隨高度變尺寸/透明度；第三人稱升俯角看深淺；三音色可辨（爆裂/悶短/脆彈）；驗收＝會打排球的人不看落點圈能接到球。

- **實作切分建議（地基級用 Fable）**：
  1. **地基級（Fable）**：D1 資料層 + 序列化、D2 比賽狀態機、D3 回合 AI 雙層狀態機、D4 Intent 管線——這四項是同一決定論骨架，當一組做，全部純函式、零 three.js、附決定論與 round-trip 測試。
  2. **手感層（Opus/一般）**：H1 五動作輸入映射、H2 視角切換運鏡、H3 欺敵曲線調參、H4 三線索呈現——接在骨架之上，靠試玩迭代。
  3. **順序**：先 D1→D2→D4（骨架）＋ D3（最難、最早開工），骨架綠燈後再疊 H 區手感，最後真機驗 60 FPS 與決定論測試。
